TCP Wave: A new reliable transport approach for future internet

发表于 2019-03-18 | 分类于 网络支撑实验室 ， 天地一体化

TCP Wave: A new reliable transport approach for future internet [2017]

观点

  • 技术发展和新的应用程序特征使得到目前为止采用的TCP 修补方法无效
    • 比如，一个改进的基于TCP的流量控制旨在实现大的拥塞窗口（cwnd）在大延迟卫星链路上，如果应用于仅限地面的链路，可能会导致过载。
  • 在采用动态交换技术（即SDN）或移动接入时，尤其强调这个问题，这会导致高度且不可预测的网络动态。

特点

  • 适应由于移动，动态切换带来的带宽/延迟变化
  • TCP Wave使用基于BURST的传输替换基于窗口的标准TCP传输，使用基于RTT的拥塞控制环（ACCE ）模块。
  • TCP Wave 设计受当前流量特性和网络演进的要求驱动
  • TCP Wave还旨在保证适应带宽和/或延迟变化，加速小型传输，快速实现高吞吐量，有效处理损失（从虚假损失到临时链路断开）等等

TCP Noordwijk：基于突发传输的先驱

  • 最初设计用于在卫星性能增强代理（PEP：Performance Enhancing Proxies）之间运行
  • BURST 大小取代了拥塞窗口（cwnd）概念，内部周期定时器，即TxTime ，调度突发传输，替代了传统TCP使用的ACK时钟定时
  • 使用ack携带的信息来估计可用带宽大小
  • TCP Noordwijk执行基于ACK的容量和拥塞估计（ACCE:ACK-based capacity and congestion estimation）功能，其监视ACK流并主动估计关于信道拥塞状态和可用容量的一组相关信息。
  • TCP Noordwijk发送方根据两种算法使用此类信息更新BURST和TxTime值：Tracking Mode和Adjustement Mode。Tracking Mode当RTT变小时运行，旨在实现最大传输速率。当RTT增长到给定阈值（β）以上时，Adjustement Mode运行，以便从即将到来的拥塞状态中恢复

  • TCP Noordwijk专门设计用于在欧洲航天局（ESA）项目框架内的PEP端之间运行，旨在提高卫星链路上的数据传输效率

    缺点

2.TCP Wave

  • 旨在成为每个宽带通信环境（包括有线地面链路）中所有标准TCP版本的有效替代方案。
  • 继承TCP Noordwijk的内容：
    • 使用TxTime来调度新的突发传输（bursts transmission）
    • 基于ACK的容量和拥塞估计（ACCE）功能块
    • 使用Tracking Mode和Adjustement Mode基本原理
  • 为了便于部署：
    • 接收端无需部署TCP Wave
    • 发送方基于标准TCP实现的设计，可以移植到普通linux上

2.3 快速重传方法

  • 与标准TCP类似，在接收到第三个dupACK时，立即重传相应的段。duplicated ACKs (dupACKs)
  • 下一个调度突发的BURST 大小减少了重新调用数据的数量，在TcpWave :: NewAck(SequenceNumber32 const& rAck)中实现

  • 使用过滤器来过滤上图两种情况：重传导致的NAck Train Dispersion远高于当前的RTT值；跟随ACK序列完全由从相同累积ACK导出的“单独”ACK组成，因此它们导致ACK列分散的计算接近于0.

  • 该算法丢弃高于测量的RTT或等于0的ACK序列分散样本

2.4 重传超时（RTO）

  • RTO通常与RTT测量成比例设置，在TCP Wave中每个突发计算一次
  • 与标准TCP不同，所有RTT测量（包括在重传期间计算的测量）都被发送到基础RTT 估计器，后者计算往返时间变化（ RTTVAR）并相应地设置RTO值。
  • RTO恢复期间TCP波发送者的行为是相当保守：发送重置BURST大小和 TXTIME为初始值 BURST_0和 TXTIME_0

2.5 Tracking Mode和Adjustement Mode模式切换

$$AvgRtt_k = αk*AvgRtt{k-1} + (1-α_k)RTT_k$$

$$α_k = {(RTT_k - minRTT_k)}/RTT_k$$

  • 当RTT接近于 在每个接收到的ACK 序列处更新的绝对最小测量值 minRTT k 时，滤波器是非常有效的。换句话说，较小的RTT样本极大地影响新的平均RTT估计，而较大的RTT样本具有较小的影响。
  • ΔRTT，用作Tracking Mode和Adjustement Mode模式切换

$$ΔRTT_k = avgRTT_k-minRTT_k$$

  • 这使得RTT跟踪更加稳定，避免了不必要的调整算法触发。给定β [s]作为参数，表示在RTT增加方面容忍的拥塞水平，而ΔRTT < β，TCP Wave保持在Tracking mode ，否则切换到Adjustement Mode

2.6 Tracking mode and “Wave” effect

  • 主要目的是实现最佳传输速率，同时考虑到瓶颈带宽和相互竞争的流的存在
  • 利用ACCE组件提供的对ACK列分散和RTT变化的连续监视，以便适当地调整传输参数。
  • TxTime在接收任何ACK列时更新为以下线性组合：
    • AckTrainDispersion，用于评估整个端到端系统处理单个burst的容量。它表示突发之间的时间间隔，以实现瓶颈链路允许的最大速率，而不依赖于并发流量的存在;
    • ΔRTT_k = avgRTT_k-minRTT_k 与即将到来的拥堵成正比。越高ΔRTT，当前的网络负载就越重。

$$TxTime = AckTrainDispersion+0.5*ΔRTT$$

  • 0.5因子是合理的，因为ΔRTT 增加当然是由于目标流量和所有竞争流量的联合效应。然后，速率降低仅占目标流量的一半。这个行动有三个目标：保证TCP Wave流之间的公平性，对非TCP Wave流友好，并保证最佳的信道利用率。
  • 为了保证最小传输速率

$$TxTime = min(TxTime,TxTime_0)$$

  • 实际传输速率

$$TxRate = BURST_0/TxTime$$

  • “Wave” effect：基于TxTime的可变调度为突发传输形成波状趋势（随着突发距离的增加减慢速率，并且获得减小距离的速率加速度）
  • 调整算法的触发代表了对网络上可能的变化的反应（链路切换，带宽变化，路由变化，或者仅存在“不受控制的”流量，例如作为UDP流），而不是简单的拥塞恢复策略。

2.7 Adjustement Mode模式

  • 当被触发时（ΔRTT > β），调整算法执行传输变量的重置，然后立即返回Tracking mode。
  • 其基本原理是允许从头开始对网络状况进行分析，而不受先前在运行连接上测量的最小RTT和平均RTT值的影响。
  • 进入Adjustement Mode有两种主要情况：
    • 持续拥挤;
    • 网络中的某些东西发生了变化; 例如，切换到具有更高延迟的链路，由于某些带宽按需MAC 协议，路由改变等导致的接入延迟增加。
  • 为了解决后者的情况不会影响前者，TCP波设想的复位minRTT和的强迫avgRTT到maxRTT值，以避免负面计算ΔRTT。该过程允许情况（2）中的TCP Wave在接收具有关于新的可能网络配置的一致值的下一个ACK列时更新minRTT 。相反，在情况（1）中，在存在持久拥塞状态的情况下，预期avgRTT 将与拥塞级别成比例地连续增长，然后返回到具有ΔRTT的跟踪模式≠0（然后以均匀下降的速率）并定期满足新触发调整模式的条件。拥塞程度越高，触发调整模式算法的频率越高。
  • 确切地说，新的调整模式算法计算突发大小等于最小突发大小（≥3 个段以保持突发结构）并且TxTime被强制为TxTime 0，以实现最小允许传输速率，目的是减少拥塞状态。

2.8 TCP Wave参数

  • 4个
    • 初始BURST大小BURST0
    • 初始传输时间（连续突发之间的间隔），TxTime 0
    • 最小允许突发大小，BURST_min ≥3 ， 范围从10到20个TCP段
    • β 阈值 β的典型值在0.1-0.3s范围内

$$Rate_{Min} = BURST_{min}/TxTime_0$$

  • 跟踪模式下的速率范围

$$BURST_{0}/(TxTime+β) <=Rate_{track} <=BURST_{0}/TxTime$$
